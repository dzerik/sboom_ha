# Документация по протоколу подключения к колонке SberBoom

## Обзор

Данный документ описывает протокол обмена данными, используемый интеграцией `sboom_ha` для Home Assistant для управления умными колонками SberBoom через локальную сеть (LAN).

---

## 1. Транспортный уровень: WebSocket over TLS (WSS)

Интеграция подключается к колонке SberBoom через **защищённый WebSocket**:

```
wss://<адрес_колонки>:20000/
```

| Параметр | Значение |
|----------|----------|
| **Протокол** | WebSocket (WSS) |
| **Порт по умолчанию** | `20000` |
| **TLS** | SSL-контекст с `verify_mode = ssl.CERT_NONE`, `check_hostname = False` |
| **User-Agent** | `WebSocket++/0.8.2` |

> **Примечание:** Колонка использует самоподписанный сертификат, поэтому проверка сертификата отключена на стороне клиента.

---

## 2. Формат сообщений: Бинарный TLV (Tag-Length-Value)

Все передаваемые данные в WebSocket — **бинарные сообщения**, закодированные в пользовательском формате TLV.

### Структура TLV

Каждое поле состоит из:
- **Ключ (байты)**: `(tag << 3) | kind` — кодируется varint
- **Значение**: зависит от `kind`

### Виды полей (kind)

| Kind | Тип | Кодирование |
|------|-----|-------------|
| `0` | varint | Целое число переменной длины (аналогично Protocol Buffers) |
| `2` | length-delimited | varint(длина) + байты (вложенные сообщения, строки, сырые данные) |

### Кодирование Varint

Varint использует 7 бит на байт для значения, старший бит (бит 7) — флаг продолжения:
- Если MSB = `1`, следуют дополнительные байты
- Если MSB = `0`, это последний байт

Пример: Значение `300` кодируется как `[0xAC, 0x02]`

### Логика декодирования

Декодер ([`_tlv.py`](../_tlv.py)) рекурсивно разбирает вложенные структуры:
1. Читает ключ varint → извлекает `tag` и `kind`
2. Для `kind=0`: читает значение varint
3. Для `kind=2`: читает длину varint, затем читает `length` байт
   - Пробует декодировать UTF-8 → если печатаемая строка, использует её
   - Пробует рекурсивное TLV-декодирование → если успешно, использует вложенный dict
   - Иначе использует hex-представление

## 3. Структура конверта сообщения

Каждый запрос, отправляемый к колонке, следует следующей структуре:

| Поле | Kind | Содержимое | Описание |
|------|------|------------|----------|
| 1 | varint | `2` | Тип сообщения: REQUEST |
| 2 | length-delimited | UUID-строка | ID запроса (для сопоставления ответов) |
| 3 | length-delimited | `pin_access_token` | Токен аутентификации |
| 5 | length-delimited | Полезная нагрузка операции | Данные запроса (вложенный TLV) |
| 6 | varint | `1` | Тип токена: PIN_AUTH |
| 7 | length-delimited | Имя клиента | Например, "Home Assistant" |
| 10 | varint | `1` | Флаг is_request |
| 11 | length-delimited | ID клиента | UUID-строка |

## 4. Процесс подключения

### 4.1 Обнаружение устройства (mDNS/Zeroconf)

Колонка транслирует своё присутствие через mDNS:

| Свойство | Значение |
|----------|----------|
| **Тип сервиса** | `_staros._tcp.local.` |
| **Свойства TXT-записи** | |
| `name` | Имя устройства (например, "SberBoom Home") |
| `type` | Модель устройства (например, "sberboom-r2") |
| `id` | Серийный/уникальный ID устройства |
| `v` | Версия прошивки |

### 4.2 Pair Handshake (Аутентификация)

Аутентификация требует **физического нажатия кнопки** на колонке.

#### Последовательность Pair

1. **Init-запрос**: Клиент отправляет `OP_PIN_CONNECT` (opcode `4`) с пустой полезной нагрузкой
   ```
   field(4, 2, field(1, 2, b""))
   ```

2. **Ответ ожидания**: Колонка отвечает статусом `1` и ID сессии
   ```json
   {5: {4: {1: 1, 2: "<id_сессии>"}}}
   ```

3. **Действие пользователя**: Пользователь нажимает кнопку `+` на колонке

4. **Ответ подтверждения**: Колонка отправляет статус `2` с `pin_access_token`
   ```json
   {5: {6: {1: 1, 2: "<pin_access_token>"}}}
   ```

5. **Сохранение токена**: Токен сохраняется в config entry для будущих подключений

#### Коды статуса Pair

| Статус | Значение |
|--------|----------|
| `1` | Ожидание нажатия кнопки `+` |
| `2` | Авторизовано, токен получен |
| `3` | Сессия уже активна (подключён другой клиент) |
| `5` | Режим Pair отключён на колонке |

#### Таймаут Pair

- **По умолчанию**: 120 секунд
- **Исключение**: `PairTimeout` при истечении времени ожидания

### 4.3 Постоянное соединение (цикл супервизора)

`SboomCoordinator` поддерживает долгоживущее соединение с автоматическим переподключением:

1. **Подключение** через WebSocket
2. **Запуск слушателя** (`_listen_loop`) для входящих сообщений
3. **Начальная синхронизация**: `get_state()` + `get_metadata()`
4. **Keep-alive**: Отправка `keep_alive()` каждые 25 секунд (настраивается)
5. **При ошибке**: Экспоненциальный backoff для переподключения

#### Последовательность Backoff при переподключении

```
1с → 2с → 5с → 10с → 30с → 60с (+ случайный джиттер до 30%)
```

#### Состояния подключения

- **Подключено**: `coordinator.connected = True`, сущности доступны
- **Отключено**: После N последовательных ошибок (настраиваемый порог)
- **Создана проблема**: Если устройство недоступно > 5 минут, создаётся проблема в Repairs

## 5. Операции (OpCodes)

### 5.1 Коды операций

| OpCode | Константа | Назначение | Полезная нагрузка запроса |
|--------|-----------|------------|---------------------------|
| `4` | `OP_PIN_CONNECT` | Pair/аутентификация | `field(1, 2, b"")` |
| `10` | `OP_GET_META_DATA` | Получить метаданные трека | `field(1, 2, b"")` |
| `12` | `OP_GET_STATE` | Получить состояние громкости/mute | `field(1, 2, b"")` |
| `14` | `OP_SET_VOLUME` | Установить громкость (0-100) | `field(1, 0, <проценты>)` |
| `15` | `OP_SET_TRACK_POS` | Перемотка на позицию | `field(1, 0, <секунды>)` |
| `16` | `OP_MEDIA_COMMAND` | Команда управления воспроизведением | `field(1, 0, <команда>)` |
| `17` | `OP_GET_PLAYING_QUEUE` | Получить очередь воспроизведения | `field(1, 2, b"")` |
| `18` | `OP_KEEP_ALIVE` | Keep-alive пинг | `field(1, 2, b"")` |

### 5.2 Медиа-команды

Используются внутри операции `OP_MEDIA_COMMAND`:

| Команда | Значение | Описание |
|---------|----------|----------|
| `MEDIA_CMD_MUTE` | `0` | Выключить звук (mute) |
| `MEDIA_CMD_UNMUTE` | `1` | Включить звук (unmute) |
| `MEDIA_CMD_NEXT` | `2` | Следующий трек |
| `MEDIA_CMD_PREV` | `3` | Предыдущий трек |
| `MEDIA_CMD_PLAY` | `4` | Начать воспроизведение |
| `MEDIA_CMD_PAUSE` | `5` | Приостановить воспроизведение |
| `MEDIA_CMD_LIKE` | `6` | Нравится текущему треку |
| `MEDIA_CMD_REMOVE_LIKE` | `7` | Убрать "нравится" |
| `MEDIA_CMD_START_MULTIROOM` | `8` | Запустить режим мультирум |
| `MEDIA_CMD_SHUFFLE_ON` | `9` | Включить перемешивание |
| `MEDIA_CMD_SHUFFLE_OFF` | `10` | Выключить перемешивание |
| `MEDIA_CMD_REPEAT_NONE` | `11` | Повтор выключен |
| `MEDIA_CMD_REPEAT_PLAYLIST` | `12` | Повтор плейлиста/всех |
| `MEDIA_CMD_REPEAT_TRACK` | `13` | Повтор одного трека |
| `MEDIA_CMD_DISLIKE` | `14` | Не нравится текущему треку |
| `MEDIA_CMD_REMOVE_DISLIKE` | `15` | Убрать "не нравится" |

## 6. Паттерны запросов/ответов

### 6.1 Fire-and-Forget (Отправил и забыл)

Используется для команд, где ответ не ожидается:
- Изменение громкости
- Операции перемотки
- Медиа-команды (play, pause, next и т.д.)
- Keep-alive

```python
await self._fire_and_forget(cast)
```

Реализация:
1. Генерация UUID ID запроса
2. Построение конверта с `with_token=True`
3. Отправка через WebSocket (под блокировкой)
4. Немедленный возврат (без ожидания ответа)

### 6.2 Request-Response (Запрос-ответ, синхронный)

Используется для запросов, где ожидается ответ:
- `get_state()`
- `get_metadata()`
- `get_queue()`

```python
resp = await self._request_response(request_data, timeout=5.0)
```

Реализация:
1. Генерация UUID ID запроса
2. Создание `asyncio.Future` и регистрация в `_pending[req_id]`
3. Построение конверта и отправка через WebSocket
4. Ожидание разрешения future (с таймаутом)
5. Цикл слушателя сопоставляет входящее сообщение по полю 2 (ID) с ожидающими future
6. Возврат байтов ответа

### 6.3 Незапрошенные Push-события

После первоначального вызова `get_metadata()` колонка начинает отправлять незапрошенные обновления:

| Поле в `req_data` | Тип события | Содержимое |
|-------------------|-------------|------------|
| `10` | Обновление метаданных | Информация о треке (название, исполнители, альбом, позиция, состояние воспроизведения) |
| `12` | Обновление состояния | Состояние громкости и mute |

> **Важно:** Изменения громкости/mute **НЕ** приходят через push-поток — они опрашиваются каждые 15 секунд (настраивается через опцию `volume_poll_interval`).

## 7. Парсинг данных

Ответы содержат **JSON-данные**, обёрнутые в бинарную TLV-обёртку.

### 7.1 Парсинг состояния

Извлекает громкость и статус mute из полезной нагрузки состояния:

```python
# Паттерн регулярного выражения
"volume":\s*\{\s*"muted":\s*(true|false)\s*,\s*"percent":\s*(\d+)
```

Результат: датакласс `SpeakerState` с полями `volume_percent` и `muted`.

### 7.2 Парсинг трека

Поддерживает два наблюдаемых формата:

#### Формат 1: Push-формат (плоский)

```json
{
  "trackId": "123456",
  "title": "Название песни",
  "artists": [{"name": "Исполнитель", "id": 789}],
  "releases": [{"id": 111, "name": "Альбом"}],
  "playing": true,
  "shuffle": false,
  "repeatType": "none",
  "position": {"tsMs": 1234567890, "val": 45},
  "duration": 180,
  "provider": "zvuk",
  "like": true,
  "explicit": false
}
```

#### Формат 2: State-формат (вложенный)

Информация о треке находится внутри объекта `info`, а статус воспроизведения — в объекте `player` на более высоком уровне:

```json
{
  "player": {
    "playing": true,
    "shuffle": false,
    "position": 45,
    "stateChangedTimestamp": 1234567890
  },
  "info": {
    "trackId": "123456",
    "title": "Название песни",
    "artists": [...],
    "duration": 180
  }
}
```

Парсер использует балансировку фигурных скобок для извлечения полного JSON-объекта, затем ищет внешний контекст `player`, если это необходимо.

### 7.3 Модель распарсенных данных

```python
@dataclass
class TrackInfo:
    title: Optional[str]           # Название трека
    artists: list[str]             # Список исполнителей
    album: Optional[str]           # Название альбома
    track_id: Optional[str]        # ID трека
    release_id: Optional[str]      # ID релиза
    artist_ids: list[str]          # ID исполнителей
    playlist_title: Optional[str]  # Название плейлиста
    provider: Optional[str]        # Провайдер (например, "zvuk")
    duration_sec: Optional[int]    # Длительность в секундах
    position_sec: Optional[int]    # Текущая позиция в секундах
    position_ts_ms: Optional[int]  # Timestamp фиксации позиции
    playing: bool                  # Воспроизводится ли сейчас
    shuffle: bool                  # Включено ли перемешивание
    repeat: Optional[str]          # Режим повтора
    explicit: bool                 # Есть ли пометка explicit
    liked: bool                    # Отмечен ли как "нравится"
    raw: dict[str, Any]            # Сырые данные
```

## 8. Архитектура интеграции Home Assistant

### 8.1 Данные Config Entry

| Ключ | Описание |
|------|----------|
| `host` | IP-адрес колонки |
| `port` | Порт WebSocket (по умолчанию: 20000) |
| `client_id` | Уникальный UUID для данного экземпляра HA |
| `client_name` | Отображаемое имя (по умолчанию: "Home Assistant") |
| `pin_access_token` | Токен аутентификации из pair handshake |
| `device_id` | Серийный номер устройства (из mDNS) |
| `device_model` | Тип модели (из mDNS) |
| `device_name` | Дружественное имя (из mDNS) |
| `device_firmware` | Версия прошивки (из mDNS) |

### 8.2 Настраиваемые опции

| Опция | По умолчанию | Описание |
|-------|--------------|----------|
| `volume_poll_interval` | 15с | Как часто опрашивать состояние громкости/mute |
| `keepalive_interval` | 25с | Интервал keep-alive пинга |
| `availability_threshold` | 3 | Последовательных ошибок до пометки "недоступно" |
| `lyrics_enabled` | True | Включить загрузку текстов песен для текущего трека |

### 8.3 События шины событий (Event Bus)

| Событие | Полезная нагрузка |
|---------|-------------------|
| `sboom_track_changed` | `track_id`, `title`, `artists`, `album`, `provider`, `previous_track_id` |
| `sboom_playback_changed` | `track_id`, `playing`, `shuffle`, `repeat` |
| `sboom_volume_changed` | `volume_percent`, `muted` |
| `sboom_connection_changed` | `connected` |

### 8.4 Платформы

| Платформа | Сущности |
|-----------|----------|
| `media_player` | Основное управление воспроизведением, громкость, выбор источника |
| `button` | Нравится, Не нравится, Следующий, Предыдущий |
| `number` | Уровень громкости (альтернативное управление) |
| `switch` | Mute, Shuffle |
| `select` | Режим повтора (none/playlist/track) |
| `sensor` | Информация о треке, позиция, длительность |
| `camera` | Обложка альбома / изображение |

## 9. Обработка ошибок

### 9.1 Типы исключений

| Исключение | Описание |
|------------|----------|
| `PairTimeout` | Истекло время ожидания pair handshake (кнопка не нажата) |
| `AuthError` | Колонка отклонила токен/UUID |
| `RuntimeError` | Не подключено / соединение потеряно |

### 9.2 Поток повторной аутентификации (Reauth)

При возникновении `AuthError`:
1. Координатор запускает `entry.async_start_reauth(hass)`
2. Config flow переходит на шаг `reauth_confirm`
3. Пользователь должен снова нажать `+` на колонке
4. Новый токен заменяет старый в config entry

### 9.3 Поток реконфигурации (Reconfigure)

Позволяет изменить host/port без повторного pair:
- Токен и device_id сохраняются
- Полезно, когда колонка получает новый IP-адрес

Обложки альбомов и другие изображения загружаются с CDN Zvuk (аутентификация не требуется):

```
https://cdn-image.zvuk.com/pic?type=<тип>&id=<id>&size=600x600
```

| Тип | Описание |
|-----|----------|
| `release` | Обложка альбома |
| `track` | Изображение конкретного трека |
| `artist` | Фото исполнителя |

## 11. Поддержка текстов песен

- **Провайдер**: Внешний сервис текстов через `lyrics_client.py`
- **Кэш**: LRU-кэш максимум на 64 трека в координаторе
- **Триггер загрузки**: При смене трека, если текст ещё не кэширован/не загружается
- **Результат**: Датакласс `Lyrics` с полями `plain` и `synced` текст, или флаг `instrumental`
